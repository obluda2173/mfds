* 6.1 Heaps

** Array-as-tree representation
- *(binary) heap*: array $A[1:n]$ viewed as a nearly complete binary tree — every level fully filled except possibly the last, which is filled left-to-right (see §B.5.3).
- Attribute $A.\text{heap-size}$ with $0 \leq A.\text{heap-size} \leq n$: only $A[1 : A.\text{heap-size}]$ count as valid heap elements; remaining slots in $A$ may hold garbage.
- $A.\text{heap-size} = 0$ → empty heap.
- Root sits at $A[1]$.

** Index arithmetic — =PARENT=, =LEFT=, =RIGHT=
#+begin_example
PARENT(i)
  return ⌊i/2⌋

LEFT(i)
  return 2i

RIGHT(i)
  return 2i + 1
#+end_example
- $\text{PARENT}(i) = \lfloor i/2 \rfloor$, $\text{LEFT}(i) = 2i$, $\text{RIGHT}(i) = 2i + 1$.
- Hardware shortcut: $2i$ is a single left bit shift, $2i + 1$ is a left shift then add 1, $\lfloor i/2 \rfloor$ is a right bit shift. Often implemented as macros or inline procedures.

** Heap property — max vs min
- Two kinds of binary heap, each defined by a *heap property* the node values satisfy.
- *Max-heap property*: for every node $i$ other than the root,
  \[ A[\text{PARENT}(i)] \geq A[i] \]
  → maximum sits at the root; every subtree rooted at $x$ contains values $\leq A[x]$.
- *Min-heap property*: for every node $i$ other than the root,
  \[ A[\text{PARENT}(i)] \leq A[i] \]
  → minimum sits at the root.
- Usage split: =HEAPSORT= uses max-heaps; min-heaps commonly back priority queues (§6.5). When a claim holds for both kinds, the source drops the prefix and just says "heap".

** Height
- *Height of a node*: number of edges on the longest simple downward path from the node to a leaf.
- *Height of the heap*: height of the root.
- A heap of $n$ elements sits on a complete binary tree → height is $\Theta(\lg n)$.
- Basic heap operations cost time proportional to the height → $O(\lg n)$.

** Core procedures (chapter roadmap)
- =MAX-HEAPIFY= — $O(\lg n)$; restores the max-heap property at one node.
- =BUILD-MAX-HEAP= — linear time; produces a max-heap from an unordered array.
- =HEAPSORT= — $O(n \lg n)$; in-place sort.
- =MAX-HEAP-INSERT=, =MAX-HEAP-EXTRACT-MAX=, =MAX-HEAP-INCREASE-KEY=, =MAX-HEAP-MAXIMUM= — priority-queue API; each $O(\lg n)$ plus the cost of mapping inserted objects to heap indices.
* 6.2 Maintaining the heap property

** =MAX-HEAPIFY= — purpose and precondition
- Inputs: array $A$ (with $\text{heap-size}$ attribute) and index $i$.
- *Precondition*: the binary trees rooted at $\text{LEFT}(i)$ and $\text{RIGHT}(i)$ are already max-heaps; only $A[i]$ may violate the max-heap property.
- *Action*: lets $A[i]$ "float down" so that the subtree rooted at $i$ obeys the max-heap property.

** Mechanism
- Determine the index of the largest value among $A[i]$, $A[\text{LEFT}(i)]$, $A[\text{RIGHT}(i)]$ (each child checked against $A.\text{heap-size}$); store it in $\textit{largest}$.
- If $A[i]$ is already largest → subtree at $i$ is a max-heap, return.
- Otherwise swap $A[i]$ with $A[\textit{largest}]$. Now $i$ and its two children satisfy the property locally, but the value just pushed into position $\textit{largest}$ may now be smaller than its own children → recurse with =MAX-HEAPIFY(A, largest)=.
#+begin_example
MAX-HEAPIFY(A, i)
  l = LEFT(i)
  r = RIGHT(i)
  if l ≤ A.heap-size and A[l] > A[i]
      largest = l
  else largest = i
  if r ≤ A.heap-size and A[r] > A[largest]
      largest = r
  if largest ≠ i
      exchange A[i] with A[largest]
      MAX-HEAPIFY(A, largest)
#+end_example

[FIGURE 6.2: Three-stage trace of =MAX-HEAPIFY(A, 2)= on a 10-node heap, $A.\text{heap-size} = 10$; violating node shown in blue at (a) $i=2$ (value 4), (b) $i=4$ after swapping with $A[4]$, (c) $i=9$ after swapping with $A[9]$, where the recursive call is a no-op.]

** Running time
- Let $T(n)$ be the worst-case time on a subtree of size at most $n$.
- Per call: $\Theta(1)$ to fix the relationship among $A[i]$, $A[\text{LEFT}(i)]$, $A[\text{RIGHT}(i)]$, plus at most one recursive call on a child's subtree.
- A child's subtree has size at most $2n/3$ (worst case occurs when the bottom level is exactly half full).
- Recurrence:
  \[ T(n) \leq T(2n/3) + \Theta(1) \tag{6.1} \]
- By case 2 of the master theorem (Theorem 4.1): $T(n) = O(\lg n)$.
- Equivalent characterization: on a node of height $h$, =MAX-HEAPIFY= runs in $O(h)$.
* 6.3 Building a heap

** =BUILD-MAX-HEAP=
- Converts $A[1:n]$ into a max-heap by calling =MAX-HEAPIFY= bottom-up.
- Elements $A[\lfloor n/2 \rfloor + 1 : n]$ are leaves → already trivial 1-element max-heaps; the loop only needs to process indices $\lfloor n/2 \rfloor$ down to $1$.
#+begin_example
BUILD-MAX-HEAP(A, n)
  A.heap-size = n
  for i = ⌊n/2⌋ downto 1
    MAX-HEAPIFY(A, i)
#+end_example

[FIGURE 6.3: Six-stage trace of =BUILD-MAX-HEAP= on $A = [4, 1, 3, 2, 16, 9, 10, 14, 8, 7]$ with $n = 10$; (a)-(e) show the state immediately before each =MAX-HEAPIFY(A, i)= call for $i = 5, 4, 3, 2, 1$ (current $i$ shown in blue), (f) shows the final max-heap.]

** Correctness — loop invariant
- *Loop invariant*: at the start of each iteration of the =for= loop, each node $i+1, i+2, \dots, n$ is the root of a max-heap.
- *Initialization*: before the first iteration, $i = \lfloor n/2 \rfloor$. Nodes $\lfloor n/2 \rfloor + 1, \dots, n$ are leaves and therefore trivial max-heap roots.
- *Maintenance*: the children of node $i$ have indices $> i$, so by the invariant both child-subtrees are max-heaps — exactly the precondition required by =MAX-HEAPIFY(A, i)= to make node $i$ a max-heap root. The call also preserves the max-heap property at nodes $i+1, \dots, n$. Decrementing $i$ then re-establishes the invariant for the next iteration.
- *Termination*: the loop runs exactly $\lfloor n/2 \rfloor$ iterations and exits with $i = 0$. The invariant then says every node $1, 2, \dots, n$ is a max-heap root — in particular the root, so $A$ is a max-heap.

** Running time
- *Loose bound*: $O(n)$ calls to =MAX-HEAPIFY=, each $O(\lg n)$ → $O(n \lg n)$. Correct but not tight.
- *Tight bound*: =MAX-HEAPIFY= on a node of height $h$ costs $O(h)$, and most nodes are near the bottom (small $h$). Two structural facts:
  - An $n$-element heap has height $\lfloor \lg n \rfloor$.
  - At most $\lceil n / 2^{h+1} \rceil$ nodes have height $h$.
- Letting $c$ be the constant inside the $O(h)$ for =MAX-HEAPIFY=, total cost is bounded by

  \begin{aligned}
    \sum_{h=0}^{\lfloor \lg n \rfloor} \left\lceil \frac{n}{2^{h+1}} \right\rceil c h
      &\leq \sum_{h=0}^{\lfloor \lg n \rfloor} \frac{n}{2^h} c h
        && (\lceil n/2^{h+1} \rceil \leq n/2^h \text{, since } \lceil x \rceil \leq 2x \text{ for } x \geq 1/2) \\
      &= c n \sum_{h=0}^{\lfloor \lg n \rfloor} \frac{h}{2^h} \\
      &\leq c n \sum_{h=0}^{\infty} \frac{h}{2^h} \\
      &\leq c n \cdot \frac{1/2}{(1 - 1/2)^2}
        && \text{(by eq. A.11 with } x = 1/2 \text{)} \\
      &= O(n)
  \end{aligned}

- *Conclusion*: a max-heap can be built from an unordered array in $\Theta(n)$ time.

** =BUILD-MIN-HEAP=
- Identical to =BUILD-MAX-HEAP= but line 3 calls =MIN-HEAPIFY= instead of =MAX-HEAPIFY=. Produces a min-heap from an unordered array in linear time.
* 6.4 The heapsort algorithm

** =HEAPSORT=
- Strategy: build a max-heap on $A[1:n]$, then repeatedly move the root (largest remaining element) to its final sorted position at the end of the array, shrink the heap, and restore the max-heap property.
- Per iteration:
  - Maximum sits at $A[1]$ → exchange $A[1]$ with $A[i]$ (current last heap slot); the largest remaining value is now in its correct final position.
  - Decrement $A.\text{heap-size}$ → node $i$ leaves the heap. The root's two subtrees are still max-heaps, but the new $A[1]$ (formerly the last heap slot) may violate the max-heap property.
  - =MAX-HEAPIFY(A, 1)= restores a max-heap on $A[1 : i-1]$.
- Repeat for heap sizes $n, n-1, \dots, 2$ (size 1 is trivially sorted).
#+begin_example
HEAPSORT(A, n)
  BUILD-MAX-HEAP(A, n)
  for i = n downto 2
    exchange A[1] with A[i]
    A.heap-size = A.heap-size - 1
    MAX-HEAPIFY(A, 1)
#+end_example

* 6.5 Priority queues

** Context
- Any comparison-based sort requires $\Omega(n \lg n)$ comparisons (Ch. 8) → =HEAPSORT= is asymptotically optimal, though a good =QUICKSORT= (Ch. 7) usually beats it in practice.
- Beyond sorting, the heap's main use is as an efficient priority queue.
- Two kinds: max-priority and min-priority queues. This section covers max; min is symmetric (Ex. 6.5-3).

** Priority queue — ADT
- *Priority queue*: data structure maintaining a set $S$ of elements, each with an associated *key*.
- *Max-priority queue* operations:
  - =INSERT(S, x, k)=: insert element $x$ with key $k$; equivalent to $S = S \cup \{x\}$.
  - =MAXIMUM(S)=: return the element with the largest key.
  - =EXTRACT-MAX(S)=: remove and return the element with the largest key.
  - =INCREASE-KEY(S, x, k)=: raise $x$'s key to $k$, assumed $\geq$ current $x.\text{key}$.
- *Min-priority queue* (symmetric): =INSERT=, =MINIMUM=, =EXTRACT-MIN=, =DECREASE-KEY=.
- Applications: job scheduling on a shared machine (max-PQ, =EXTRACT-MAX= picks next job); event-driven simulation (min-PQ, =EXTRACT-MIN= picks next event by time). =DECREASE-KEY= features prominently in Ch. 21 and 22.

** Mapping application objects ↔ heap indices
- Heap slots store pointers to application objects; each object carries a =key= attribute, so $A[i].\text{key}$ is well-defined.
- Heap positions change during operations, so both directions of the mapping (object → index, index → object) must be maintainable.
- *Handles*: opaque tokens stored in both the object and the heap element, giving enough information to translate. When a heap element moves, the handle in the corresponding object is updated. Typical worst-case overhead: $O(1)$ per access.
- *Alternative*: store the object → index mapping inside the priority queue itself.
  - Advantage: application objects need no modification.
  - Disadvantage: cost of establishing and maintaining the mapping.
  - Hash-table option (Ch. 11): expected $O(1)$ per lookup, worst case $\Theta(n)$.
- =MAX-HEAPIFY= is implicitly assumed to compare by =.key=, exchange pointers (not values), and update the index mapping on every swap. All running times below are $\cdot + \text{mapping overhead}$.

** =MAX-HEAP-MAXIMUM= — implements =MAXIMUM=
- Root of a max-heap is the maximum → return $A[1]$ after an underflow check.
- *Running time*: $\Theta(1)$.
#+begin_example
MAX-HEAP-MAXIMUM(A)
  if A.heap-size < 1
      error "heap underflow"
  return A[1]
#+end_example

** =MAX-HEAP-EXTRACT-MAX= — implements =EXTRACT-MAX=
- Save $A[1]$, move the last heap element into the root, shrink the heap, then =MAX-HEAPIFY(A, 1)= to restore the property. Structurally identical to the body of the =HEAPSORT= for-loop (lines 3–5).
- *Running time*: $O(\lg n)$ — constant work on top of =MAX-HEAPIFY=, plus mapping overhead.
#+begin_example
MAX-HEAP-EXTRACT-MAX(A)
  max = MAX-HEAP-MAXIMUM(A)
  A[1] = A[A.heap-size]
  A.heap-size = A.heap-size - 1
  MAX-HEAPIFY(A, 1)
  return max
#+end_example

** =MAX-HEAP-INCREASE-KEY= — implements =INCREASE-KEY=
- Reject if the new key $k$ would *decrease* $x.\text{key}$, then set $x.\text{key} = k$.
- The increase may break the max-heap property along the path from $x$ toward the root. Walk that path: while the current node's key exceeds its parent's, swap with the parent (updating the index mapping); stop once parent $\geq$ current (property restored) or root is reached.
- Structure mirrors the inner loop of =INSERTION-SORT= (p. 19).
- *Running time*: $O(\lg n)$ — the path to the root has length $O(\lg n)$.
#+begin_example
MAX-HEAP-INCREASE-KEY(A, x, k)
  if k < x.key
      error "new key is smaller than current key"
  x.key = k
  find the index i in array A where object x occurs
  while i > 1 and A[PARENT(i)].key < A[i].key
      exchange A[i] with A[PARENT(i)], updating the information that maps
              priority queue objects to array indices
      i = PARENT(i)
#+end_example

** =MAX-HEAP-INSERT= — implements =INSERT=
- Check capacity ($A.\text{heap-size} < n$), extend the heap by one slot whose key is the sentinel $-\infty$, map $x$ to that index, then call =MAX-HEAP-INCREASE-KEY= to raise the key to its true value $k$. The $-\infty$ sentinel makes the =INCREASE-KEY= precondition $k \geq x.\text{key}$ hold trivially.
- *Running time*: $O(\lg n)$ plus mapping overhead.
#+begin_example
MAX-HEAP-INSERT(A, x, n)
  if A.heap-size == n
      error "heap overflow"
  A.heap-size = A.heap-size + 1
  k = x.key
  x.key = -∞
  A[A.heap-size] = x
  map x to index heap-size in the array
  MAX-HEAP-INCREASE-KEY(A, x, k)
#+end_example

** Summary
- A heap supports every priority-queue operation on a set of size $n$ in $O(\lg n)$ time, plus the overhead for mapping objects to array indices.
